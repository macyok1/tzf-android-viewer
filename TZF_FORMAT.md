# Исследование формата TZF

Этот документ фиксирует только проверенные факты и явно отделяет их от гипотез.

## Тестовый файл

- Файл: `Scan_001.tzf`
- Размер: `54 546 415` байт
- Источник эталонных данных: Trimble RealWorks 12.0
- Известная сетка: `1742 x 4123`
- Известное число валидных точек: `5 524 662`

## Каталог блоков по адресу `0x2690F88`

Уверенность: высокая. Поля прочитаны непосредственно из TZF, а offsets нескольких блоков подтверждены ETW-трассой RealWorks.

```text
+0x00  8 bytes   неизвестная сигнатура/идентификатор
+0x08  u32 LE    200
+0x0C  u32 LE    1764 — размер payload
+0x10  u32 LE    4 — число компонентов (первое поле payload)
+0x14             начало component[0]
```

Payload полностью описывается формулой:

```text
u32 componentCount
repeat componentCount times:
    u32 componentId
    u32 descriptorCount
    descriptor descriptors[descriptorCount]
```

Дескриптор занимает 12 байт:

```text
+0x00  u64 LE    offset блока в TZF
+0x08  u32 LE    размер/длина блока
```

В тестовом файле четыре компонента с ID `1..4`; каждый содержит 36 дескрипторов. Размер структуры совпадает до байта: `4 + 4 x (8 + 36 x 12) = 1764`. Все 144 дескриптора проверяются относительно границ TZF.

36 блоков являются тайлами `512 x 512`: `ceil(1742 / 512) x ceil(4123 / 512) = 4 x 9 = 36`. Индекс тайла вычисляется как `tileX * 9 + tileY`. Для строки скана 1000 `tileY = 1`, поэтому `TZF_GetXYZ` читает тайлы `1, 10, 19, 28` компонентов 1–3. Это подтверждено ETW-трассой.

Первые три дескриптора:

```text
0: offset=0x16FEF8, size=524358
1: offset=0x659E6F, size=654719
2: offset=0xB470E9, size=814328
```

Offsets `0x16FEF8`, `0x659E6F`, `0x4AD314`, `0x58FD14` и `0x8F38D7` непосредственно присутствуют в сохранённой ETW-карте чтений. Остальные блоки могли не читаться при запросе одного столбца.

## Заголовок тайла

Уверенность: высокая для смещений и размеров полей; назначение scale и codecId подтверждается дополнительно.

```text
+0x00  u32 LE    marker = 100
+0x04  u32 LE    width = 512
+0x08  u32 LE    height = 512
+0x0C  u32 LE    размер распакованного канала
+0x10  f32 LE    масштаб: 10.0, 1000000.0 или 1.0
+0x14  u64 LE    идентификатор кодека
+0x1C             начало сжатого payload
```

Компоненты 1–3 имеют распакованный размер `512 x 512 x 4 = 1 048 576` байт. Компонент 4 имеет размер `512 x 512 = 262 144` байт и его payload начинается с JPEG/JFIF. Следовательно, компонент 4 хранит одноканальное изображение, вероятно интенсивность или яркость цвета; точное назначение требует сравнения с RealWorks.

Идентификаторы на тестовом файле:

```text
component 1: 0x839A721B429840B9, scale 10.0
component 2: 0xAFEC6C11BC26B6C5, scale 1000000.0
component 3: 0xAFEC6C11BC26B6C5, scale 1000000.0
component 4: 0xB2EBC9519992D032, scale 1.0, JPEG/JFIF
```

Сопоставление с экспортами `GetGUID` из `ZZCodecs_x64.dll`:

```text
0x839A721B429840B9 = SnappyChunkCodec
0xAFEC6C11BC26B6C5 = Snappy_Transpose_DeriveChunkCodec
0xB2EBC9519992D032 = JPEGChunkCodec
```

`Snappy_Transpose_DeriveChunkCodec::_Decode` выполняет:

1. raw Snappy decode;
2. накопительную сумму `uint32` по всему массиву с 32-битным переполнением;
3. транспонирование: `dst[y * width + x] = src[x * height + y]`.

Порядок восстановлен непосредственно по машинному коду DLL и воспроизведён в `tzf-core` без использования DLL во время работы.

## Исправленная гипотеза

Предыдущая рабочая сессия трактовала `1764` как число 36-байтных записей. Проверка дала выходящие за файл offsets уже с предполагаемой записи 12. Полный разбор payload доказал, что `1764` является его размером, `4` — числом компонентов, а `36` внутри каждого компонента — числом дескрипторов.
