# Точная constrained-регистрация двух TZF-сканов

## Цель

Добавить автоматическое уточнение предварительно совмещённых вручную облаков. Пользователь грубо располагает второй скан с помощью gizmo, после чего приложение выполняет многоуровневую регистрацию и изменяет только X, Y, Z и RZ.

Результат применяется только при доказанном качестве. Низкое внутреннее значение residual само по себе не считается гарантией миллиметровой геодезической точности.

## Границы

В этот этап входят:

- native constrained point-to-plane ICP;
- фоновая сессия регистрации и отмена;
- проверка результата и численные метрики;
- отдельная команда «Точная сшивка»;
- переименование команды камеры в «Показать всё»;
- удаление ручных числовых полей, шагов и кнопок `+/-` из меню сшивки.

В этот этап не входят интерактивный куб вида, экспорт ASC и Wi-Fi-передача со сканера. Они проектируются и реализуются отдельно.

## Ограничение трансформации

Оптимизируются ровно четыре параметра:

- translation X;
- translation Y;
- translation Z;
- rotation Z вокруг мирового вертикального направления.

RX и RY отсутствуют в параметрах решателя и не вычисляются даже как промежуточный результат. Если unconstrained 6-DoF-преобразование могло бы дать меньший residual, оно всё равно запрещено.

Текущая трансформация gizmo служит начальным приближением. Регистрация не заменяет грубый глобальный поиск положения.

## Данные регистрации

Preview до 150 000 точек используется только для изображения и ручного приближения. Native-код повторно читает TZF и строит собственные пространственно равномерные выборки большей плотности.

Для обоих облаков создаются несколько уровней разрешения:

1. грубый уровень для устойчивого захвата;
2. средний уровень для уточнения;
3. финальный уровень для локальной точности;
4. отдельная плотная контрольная выборка для проверки результата.

Размер voxel и число точек выбираются из пространственного размера, плотности скана и доступной памяти. Выборка должна покрывать весь доступный объём, а не первые тайлы TZF.

Перед заявлением порогов в миллиметрах требуется подтвердить коэффициент единиц TZF по метаданным либо внешнему эталонному расстоянию. До этого значения отображаются как единицы скана.

## Алгоритм

Основой служит многоуровневый point-to-plane ICP с четырёхпараметрическим обновлением XYZ+RZ.

На каждом уровне:

- для опорного облака оцениваются локальные нормали;
- для точек перемещаемого облака находятся ближайшие соответствия;
- пары фильтруются по максимальному расстоянию и совместимости нормалей;
- point-to-plane residual взвешивается robust loss;
- решается малое обновление X, Y, Z и RZ;
- проверяются уменьшение ошибки, обусловленность и размер шага;
- цикл останавливается при сходимости либо превышении лимита итераций.

Решатель не должен сходиться ценой нескольких повторяющихся плоскостей или симметричного коридора. Плохая обусловленность является причиной отказа.

## Проверка результата

Итоговая трансформация проверяется на отдельной плотной выборке. Вычисляются:

- RMS расстояний принятых соответствий;
- 95-й процентиль расстояний;
- доля контрольных точек с допустимым соответствием;
- пространственное распределение соответствий;
- обусловленность финального решения;
- число использованных точек и итераций.

По умолчанию после подтверждения метрических единиц используются пороги RMS 3 мм и P95 8 мм. Оба порога настраиваются пользователем. Минимальная доля и пространственное покрытие задаются алгоритмом и не скрываются из отчёта.

Трансформация принимается только при прохождении всех проверок. При недостаточном перекрытии, неоднозначной геометрии, расходимости или превышении порогов исходная ручная трансформация остаётся без изменений.

## Архитектура

### Native registration core

Чистый C++-модуль отвечает за выборки, spatial index, нормали, соответствия, constrained solver, robust loss и метрики. Он не зависит от Android UI и запускается portable-тестами на Windows/Linux CI.

### RegistrationSession

Java-объект владеет generation ID, путями двух локальных TZF, исходной XYZ+RZ-трансформацией, настройками качества, прогрессом и отменой. Декодирование нового файла или повторный запуск инвалидирует старую сессию.

### JNI

JNI передаёт четыре исходных параметра, настройки и сигнал отмены. Native-код публикует компактный прогресс по этапам и возвращает либо проверенный результат с метриками, либо типизированную причину отказа.

### Renderer и UI

Renderer не выполняет оптимизацию. После успеха он атомарно применяет новую четырёхкомпонентную трансформацию. UI синхронизирует summary и gizmo через существующий listener.

## Интерфейс

Верхняя команда «Вписать» переименовывается в «Показать всё» и продолжает управлять только камерой.

В режиме сшивки появляется отдельная основная кнопка «Точная сшивка». Она активна только после загрузки двух облаков.

Из меню сшивки полностью удаляются:

- числовые поля X, Y, Z и RZ;
- поля шага;
- кнопки `+/-`;
- связанная логика парсинга и рекурсивной синхронизации.

Ручное предварительное совмещение выполняется gizmo. Текстовый summary XYZ+RZ может остаться как read-only диагностика.

Во время регистрации показываются этап, процент, количество соответствий и кнопка отмены. После завершения выводятся RMS, P95, доля совпадений и статус «Принято» либо конкретная причина отказа.

Настройки качества содержат пороги RMS и P95. Пока единицы не подтверждены, интерфейс не подписывает их как миллиметры.

## Ошибки и безопасность состояния

Ни одна ошибка не должна частично изменять текущую трансформацию. К причинам отказа относятся:

- недостаточное перекрытие;
- слишком далёкое начальное положение;
- вырожденная или симметричная геометрия;
- недостаток корректных нормалей;
- расходимость или плохая обусловленность;
- превышение RMS/P95;
- нехватка памяти;
- отмена или устаревшая generation-сессия;
- ошибка чтения TZF.

Последний корректный ручной результат сохраняется. `NaN`, infinity и RX/RY не могут попасть в renderer или сохранённое состояние.

## Проверка

### Synthetic C++ tests

- известные XYZ+RZ-преобразования;
- несколько уровней шума и выбросов;
- частичное перекрытие;
- плоскости, углы, коридоры и неоднозначная геометрия;
- отказ при плохом начальном приближении;
- строгая неизменность RX/RY;
- отмена между итерациями.

### Реальные данные

Нужны минимум две перекрывающиеся TZF-станции и эталонная трансформация из RealWorks/FARO либо измеренные контрольные марки. На них сравниваются XYZ+RZ и независимые контрольные ошибки.

### Android

На Samsung SM-G998B проверяются время, пиковая память, отзывчивость UI, отмена, повторный запуск, смена ориентации и длительный расчёт. После каждого сценария проверяется `logcat`.

### CI

GitHub Actions запускает JVM-тесты, portable C++ registration tests и сборку debug APK.

## Критерии готовности

- Команда «Точная сшивка» действительно запускает регистрацию, а «Показать всё» управляет только камерой.
- Оптимизация использует полные TZF-данные и изменяет только XYZ+RZ.
- Ненадёжный результат никогда не применяется.
- Метрики вычисляются на отдельной контрольной выборке.
- Пороги RMS/P95 настраиваются и по умолчанию равны 3/8 мм после подтверждения единиц.
- Ручные числовые поля и кнопки шагов удалены.
- Результат проверен на synthetic tests, реальных эталонных данных, Samsung и CI.
- Заявление о миллиметровой точности делается только по независимым контрольным измерениям.
