# Perspective-процесс сканирования, сшивки и управления камерой

## Статус

Дизайн согласован 2026-07-15. Выбран контекстный интерфейс: цель сшивки задаётся перед запуском X7, а R/M и манипулятор существуют только внутри ручной сшивки.

## Цель

Перестроить полевой процесс по модели Trimble Perspective:

- один палец по умолчанию двигает сцену в плоскости экрана;
- вращение камеры включается отдельной кнопкой справа;
- новый скан X7 после скачивания автоматически сшивается с выбранной станцией;
- успешные парные связи формируют явные наборы регистрации;
- неудача не останавливает дальнейшее сканирование и предлагает ручную сшивку;
- ручной вид сверху и временные R/M открываются только по явной команде.

## Исследованный механизм Perspective

Локальная документация `C:\persp\Help\Web\en` подтверждает следующие правила Perspective:

- новый скан по умолчанию регистрируется с предыдущим, но reference можно выбрать из списка `Register to`;
- успешная регистрация создаёт направленную связь moving → reference;
- цепочка успешно связанных станций образует один registration set;
- ошибка оставляет станцию в новом отдельном registration set;
- ручная регистрация показывает временные списки Moving и Reference и кольцевой манипулятор;
- перемещение станции из registration set в ручном режиме перемещает набор как жёсткое целое.

Эти правила переносятся как модель данных и состояний, а не только как внешний вид.

## Границы

### Входит

- режимы камеры PAN и ORBIT;
- правая кнопка переключения режима;
- выбор reference перед запуском X7;
- автоматический запуск регистрации после успешного скачивания TZF;
- регистрационные наборы, связи и статусы;
- отдельное представление дерева по папкам и по наборам сшивки;
- транзакционный ручной режим сверху;
- миграция старых проектов;
- JVM, native-regression и X7 workflow-тесты.

### Не входит

- изменение уже рабочего native-алгоритма Perspective-style registration;
- глобальная оптимизация полного pose graph;
- автоматическая сшивка произвольно импортированных локальных TZF;
- разрыв и редактирование подтверждённых связей;
- превращение пользовательских папок в жёсткие регистрационные объекты;
- удаление существующих пользовательских папок и их вложенности.

## Управление камерой

### Режимы

`PointCloudView` получает явный `CameraGestureMode`:

- `PAN` — режим по умолчанию после каждого открытия проекта;
- `ORBIT` — временный режим вращения камеры.

Режим не сохраняется в проекте и после перезапуска всегда возвращается в `PAN`.

### Жесты

- Один палец в `PAN` вызывает существующее экранное панорамирование `panByPixels`. Движение идёт в плоскости экрана при любом ракурсе; сверху оно совпадает с мировой XY.
- Один палец в `ORBIT` изменяет yaw/pitch текущей камеры.
- Щипок двумя пальцами изменяет только zoom. Старое дополнительное перемещение по средней точке двух пальцев удаляется.
- Куб ориентации, готовые ракурсы и переключение перспективной/ортографической проекции остаются.
- Приоритет специализированных инструментов сохраняется: clip handle, измерение и ручной registration gizmo обрабатывают жест раньше камеры.

### Кнопка

Справа над viewport добавляется кнопка `↻` с touch target не меньше 48 dp:

- нейтральная поверхность — PAN;
- голубой фон/контур — ORBIT;
- `contentDescription` сообщает активный режим и действие следующего нажатия;
- внутри ручной сшивки кнопка скрыта, потому что камера зафиксирована сверху.

## Доменная модель регистрации

Формат проекта повышается с v4 до v5.

### `RegistrationSet`

Явный системный объект:

- стабильный UUID;
- отображаемое имя `Набор регистрации N`;
- упорядоченный список ID станций;
- дата создания и последнего изменения.

Каждая станция принадлежит ровно одному registration set. Набор не является `ProjectModel.Group` и не влияет на пользовательские папки.

### `RegistrationLink`

Подтверждённая направленная связь содержит:

- стабильный UUID;
- ID moving-станции;
- ID reference-станции;
- относительный XYZ+RZ-transform moving в системе reference после применения, определённый формулой `movingWorld = compose(referenceWorld, relative)`;
- RMS, P95, overlap, consistency и confidence;
- источник `AUTO_X7` или `MANUAL`;
- время подтверждения.

Связь создаётся только после принятого результата. `CHECK` и `FAILED` связей не создают.

### Состояние станции

Станция хранит:

- `UNLINKED` — новая или первая станция без подтверждённой связи;
- `PENDING` — TZF готов и ожидает автоматическую/повторную сшивку;
- `REGISTERED` — существует подтверждённая связь;
- `CHECK` — найден кандидат, требующий подтверждения;
- `FAILED` — попытка завершилась без принятой связи;
- `LEGACY_UNLINKED` — размещение перенесено из проекта v4 без придуманной связи.

Дополнительно сохраняются ID последней attempted reference, диагностическая причина и последние метрики. Для `CHECK` сохраняется прямой world transform кандидата XYZ+RZ, уже переведённый из native pivot-конвенции; это позволяет подтвердить или отклонить тот же результат после перезапуска. `REGISTERING` остаётся только runtime-состоянием; после аварийного перезапуска оно читается как `PENDING`.

## Операции с наборами

### Создание

Каждый скачанный скан сначала получает собственный новый registration set. Первый скан проекта остаётся `UNLINKED` и служит допустимой опорой для следующего.

### Успешная регистрация

Reference set остаётся неподвижным. Для moving-станции вычисляется жёсткая correction `candidateMovingWorld ∘ inverse(oldMovingWorld)`. Для каждой станции moving set применяется `newWorld = correction ∘ oldWorld`, поэтому их внутренние взаимные положения сохраняются. Затем:

1. создаётся `RegistrationLink`;
2. участники moving set переносятся в reference set;
3. пустой moving set удаляется;
4. проект атомарно сохраняется.

Это позволяет позднее сшить не только один неудачный скан, но и построенную от него последующую цепочку.

### Ошибка и CHECK

- `FAILED` оставляет transform и registration set без изменений.
- `CHECK` показывает кандидат, но не двигает проект и не объединяет наборы до подтверждения.
- Отказ от кандидата переводит станцию в `FAILED`.

## Выбор «Сшить с»

Выбор расположен в карточке запуска X7 перед кнопкой `Начать скан`.

- Значение по умолчанию — динамический пункт `Последний: <имя>`.
- `Последний` означает последний успешно скачанный скан, включая станцию, чья предыдущая регистрация завершилась ошибкой. Это сохраняет Perspective-цепочку: после разрыва новые станции могут сформировать отдельный рабочий набор.
- Остальные варианты — все доступные скачанные станции по убыванию времени получения.
- Первый скан не имеет reference, поэтому selector выключен.
- При нажатии `Начать скан` alias разрешается в конкретный station ID и фиксируется для этой операции. Изменение списка позднее не меняет in-flight target.
- Удалённая или недоступная станция исчезает из вариантов; до запуска выбирается новый последний доступный скан.

## X7 workflow

Отдельный `RegistrationWorkflow` координирует существующие `TrimbleX7Client`, скачивание, загрузку сцены и native registration. `MainActivity` только отображает состояние и передаёт пользовательские команды.

Последовательность:

1. пользователь выбирает `Сшить с` и запускает X7;
2. target station ID фиксируется в capture session;
3. существующий X7-процесс выполняет scan и download;
4. валидный TZF добавляется в проект и немедленно появляется в списке станций;
5. после готовности preview новый скан получает `PENDING` и отдельный registration set;
6. если reference существует, workflow запускает automatic registration;
7. `accepted global` создаёт связь и объединяет наборы;
8. `check registration` создаёт незафиксированный кандидат и состояние `CHECK`;
9. отказ native, отмена или недостаточные метрики дают `FAILED` и кнопку `Сшить вручную`.

Ошибка регистрации не считается ошибкой сканирования или скачивания. Пользователь может сразу запустить следующий скан.

Обычный импорт TZF и команда загрузки уже готового файла с X7 только добавляют станцию; автоматический post-download registration запускается исключительно capture session, начатой кнопкой `Начать скан`.

## Представление станций

Панель дерева получает переключатель:

- `Папки` — существующая пользовательская иерархия без изменений;
- `Наборы сшивки` — read-only группировка станций по `RegistrationSet`.

Карточка станции показывает компактный статус:

- зелёный `Сшит`;
- оранжевый `Проверить`;
- красный `Не сшит`;
- серый `Без связи`.

У `FAILED` основное действие — `Сшить вручную`. Постоянная карточка R/M и постоянно активный манипулятор удаляются из обычного workspace.

## Ручная сшивка

### Вход

Режим открывается только по `Сшить вручную` или контекстному действию станции. По умолчанию:

- M — выбранная станция и весь её registration set;
- R — attempted reference, а при её отсутствии предыдущая скачанная станция из другого набора.

Списки R/M существуют только в toolbar ручного режима. R и M не могут принадлежать одному registration set.

### Транзакция

Перед первым изменением сохраняются world transforms всех станций moving set. Затем:

- viewport переключается строго на ортографический вид сверху;
- маркер перемещает набор M по XY;
- кольцо вращает набор M вокруг выбранной moving-станции;
- щипок двумя пальцами продолжает изменять масштаб, не меняя зафиксированный верхний ракурс;
- внутренние transforms станций набора не меняются;
- `Сшить` запускает constrained local registration от ручной позиции.

Успешный native-результат показывается как кандидат с метриками. `Применить` создаёт link, объединяет наборы и завершает сессию. `Отмена` на любом этапе восстанавливает snapshot всех world transforms и не меняет связи.

Если local registration отклонён, пользователь остаётся в ручном режиме с исходной ручной позицией и диагностикой; автоматического выхода или скрытого commit нет.

## Компоненты

### `RegistrationGraph`

Чистый Java-компонент без Android-зависимостей:

- создание наборов;
- проверка единственности membership;
- merge moving set → reference set;
- создание и валидация link;
- применение rigid correction ко всему moving set;
- поиск предыдущей и последней станции;
- миграция v4 → v5.

### `RegistrationWorkflow`

Оркестратор состояний:

- snapshot выбранного target;
- post-download auto registration;
- generation/cancellation;
- success/check/failure transitions;
- восстановление `PENDING` после перезапуска;
- публикация immutable UI state.

### `ManualRegistrationSession`

Владеет R/M, transform snapshot, временным preview и commit/rollback. Сессия не сериализует промежуточный drag.

### UI

- `PointCloudView` отвечает только за жесты, camera mode и временный gizmo transform;
- `ScanTreePanel` отображает папки либо registration sets;
- `MainActivity` связывает lifecycle, workflow и существующие панели, не реализуя graph-операции самостоятельно.

## Миграция проектов

При чтении v4:

- все URI, папки, visibility и XYZ+RZ-transforms сохраняются без изменений;
- каждый существующий скан получает отдельный registration set;
- состояние становится `LEGACY_UNLINKED`;
- фиктивные registration links не создаются;
- первое успешное новое или ручное связывание объединяет соответствующие наборы обычным способом.

Это может показать больше отдельных наборов, но не приписывает старому проекту несуществующие доказанные связи.

## Ошибки и восстановление

- Сбой scan/download не создаёт станцию и не запускает registration.
- Сбой registration не удаляет скачанный TZF и не меняет transforms.
- Потеря процесса во время `REGISTERING` восстанавливается как `PENDING` с явной кнопкой `Повторить`.
- Автоповтор после запуска приложения не выполняется бесконечно и не начинается без команды пользователя.
- Удаление reference до регистрации даёт понятный `FAILED: reference недоступна`.
- Устаревшая generation не может опубликовать transform, link или set merge.
- Сохранение проекта выполняется после целиком завершённой доменной операции; частичный merge не сериализуется.

## Проверка

### JVM

- PAN является режимом по умолчанию, ORBIT не сохраняется;
- one-finger dispatch для PAN/ORBIT и приоритет gizmo/clip/measure;
- pinch изменяет zoom без midpoint pan;
- создание набора для новой станции;
- merge сценариев `1+2`, `12+3`, `4+5`, `123+45`;
- rigid correction moving set без изменения внутренних относительных transforms;
- запрет R/M из одного набора;
- динамический и зафиксированный `Последний`;
- переходы success/check/failed/pending;
- rollback ручной сессии;
- миграция v4 без изменения world transforms;
- round-trip codec v5 для sets, links и метрик.

### Native regression

Native API не меняется. Повторно проверяются:

- синтетические constrained/global tests;
- реальная TDX-пара с известным transform;
- ручная позиция с ошибкой и заведомо неверная позиция;
- единый centroid pivot между Java и C++.

### X7 и Android

- первый скан без reference;
- второй скан с default `Последний`;
- явный выбор более старой станции;
- успешный scan → download → registration → set merge;
- CHECK без скрытого link;
- FAILED с продолжением сканирования и ручной кнопкой;
- kill/reopen после download и во время registration;
- ручная отмена с полным восстановлением moving set;
- portrait, landscape, phone и tablet для правой кнопки и обеих форм дерева.

## Этапы реализации

1. Изолированно внедрить PAN/ORBIT и тесты жестов.
2. Добавить v5 model, codec, `RegistrationGraph` и миграцию.
3. Добавить отображение registration sets и статусов без изменения X7.
4. Встроить selector и post-download `RegistrationWorkflow` в X7 capture.
5. Перевести manual registration на транзакционный `ManualRegistrationSession` и убрать постоянные R/M.
6. Прогнать реальные TZF/X7 проверки, собрать APK и только после этого публиковать nightly.

## Критерии готовности

- Обычный свайп одним пальцем всегда панорамирует, пока пользователь явно не включил ORBIT.
- Новый X7 scan использует зафиксированную перед стартом reference и появляется в списке сразу после скачивания.
- Успех создаёт link и объединяет sets; CHECK и FAILED этого не делают.
- Ошибка не запускает ручной режим автоматически и не блокирует следующий scan.
- Ручная отмена не оставляет частично перемещённых станций.
- R/M и кольцевой манипулятор отсутствуют вне ручной сшивки.
- Пользовательские папки и transforms старых проектов сохраняются.
- Все JVM/native/CI проверки проходят, а реальный nightly публикуется только после проверки полного цикла.
